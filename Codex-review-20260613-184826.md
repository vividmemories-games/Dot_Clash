## Findings

### [P1] Derive challenge result from the room winner

File: `/Users/t947967/Documents/Personal Projects 2/Dot_Clash/lib/features/challenge/presentation/challenge_game_bindings.dart`

Lines: 116-122

For normal finished challenges, the UI/analytics outcome is derived from the local `challengeGameProvider` state. The terminal room snapshot can arrive before that notifier has applied the final board, so the dialog can show the wrong winner even though the server settlement is correct. Use `room.winnerUid` as the single source of truth for both `finished` and `abandoned` states.

### [P2] Settle already-terminal rooms

File: `/Users/t947967/Documents/Personal Projects 2/Dot_Clash/lib/features/challenge/presentation/challenge_game_bindings.dart`

Lines: 54-59

This listener only reacts to future room changes. If the app is killed before settlement and later reopens `/challenge/play/{code}` after the room is already `finished` or `abandoned`, `_settle` never runs, so the user may never get their server-side match/stat settlement. Add `fireImmediately: true` or explicitly settle the current terminal room while keeping the dialog behavior guarded.

### [P2] Gate local match-session listeners in challenge mode

File: `/Users/t947967/Documents/Personal Projects 2/Dot_Clash/lib/features/game/presentation/game_screen.dart`

Lines: 325-338

The challenge screen still listens to `matchSessionProvider`, so stale campaign/AI session flags can show out-of-turns/riposte UI during a live challenge. The same provider is also read by `canInteract`, which can disable the challenge board if `outOfTurnsPending` leaks through. Skip this listener and use a neutral session path when `_isChallenge` is true.

### [P2] Restrict targeted challenge pushes

File: `/Users/t947967/Documents/Personal Projects 2/Dot_Clash/functions/src/challenge.ts`

Lines: 257-258

Any signed-in caller who knows a UID can pass it as `targetUid` and trigger challenge push notifications, with no recent-rival check or rate limit. Since the product scope says targeted pushes are for recent rivals, enforce that relationship server-side and/or add throttling before sending FCM.

### [P2] Challenge settlement now mutates economy/progression

File: `/Users/t947967/Documents/Personal Projects 2/Dot_Clash/functions/src/challenge.ts`

Lines: 454-507

The plan explicitly keeps challenge v1 out of lives/coins/XP/economy, but this callable awards coins and XP, changes rating, season stats, lives state, and profile counters. If the plan is still authoritative, limit this to challenge match history/H2H fields or update the release plan and balance/security expectations accordingly.
